# 谁家的小毛咪｜开发可读状态机文档

## 1. 文档目的

将原始“剧情树”与已设计的 UI 页面流改写为开发可读的状态机结构，便于后续交给 AI 或开发者生成小程序页面、组件、交互逻辑、弹窗逻辑、音频播放逻辑与分享海报逻辑。

## 2. 状态机字段规范

| 字段 | 说明 |
|---|---|
| `id` | 节点唯一 ID，建议前缀区分模块 |
| `parentId` | 父级节点 ID，根节点为 `null` |
| `nodeType` | 节点类型：`ROOT`、`SCENE`、`PROMPT`、`OPTION_VALID`、`OPTION_INVALID`、`AUDIO_OPTION`、`RESULT_MODAL` |
| `text` | 页面展示文案或弹窗文案 |
| `isClickable` | 是否可点击。`false` 不代表无反馈，可由前端做“无效点击拦截反馈” |
| `triggerFeedback` | 点击或进入该状态时触发的反馈，如弹窗、音频、截图分享、重新选择 |
| `nextId` | 点击后进入的下一节点；若为终点则为 `END` |
| `assetNeeds` | 所需素材，如声音、icon、插画、弹窗按钮、分享海报模板 |

## 3. 通用交互规则

### 3.1 节点类型说明

| 节点类型 | 开发含义 |
|---|---|
| `ROOT` | 小程序进入后的根状态 |
| `SCENE` | 剧情段落或大的交互章节 |
| `PROMPT` | 当前页面问题/引导语 |
| `OPTION_VALID` | 有效选项，点击后触发正向反馈或进入下一状态 |
| `OPTION_INVALID` | 无效选项，视觉上可置灰/弱化；点击时触发惩罚式弹窗或重选反馈 |
| `AUDIO_OPTION` | 点击后需要播放声音，并进入下一层选择 |
| `RESULT_MODAL` | 弹窗结果态，通常为终点或返回上一步 |

### 3.2 “不可点击但点击跳弹窗”的处理方式

原框架中多次出现“不可点击，点击跳弹窗”。开发上建议统一处理为：

```ts
isClickable: false
invalidTapFeedback: showModal(...)
```

即：

- 视觉上表现为不可选/被拒绝/弱化按钮；
- 但前端仍监听 tap 事件；
- 点击后不进入主线状态，只弹出提示；
- 弹窗关闭后回到当前父级问题。

### 3.3 成功弹窗通用配置

| 配置项 | 内容 |
|---|---|
| 弹窗文案 | `耶斯！小毛咪不生气啦` |
| 弹窗按钮 | `截图分享给姐姐` |
| 按钮行为 | 生成当前结果截图/分享海报 |
| 终点 | `END_SUCCESS_SHARE` |

### 3.4 重选弹窗通用配置

| 配置项 | 内容 |
|---|---|
| 弹窗文案 | 根据节点变化，如 `哎呀！被亲亲了！，重选一次吧` |
| 弹窗按钮 | `请重新选择` 或关闭弹窗后回到父级 |
| 按钮行为 | 返回当前父级问题 |
| 终点 | 非终点 |

# 4. UI 界面设计补充规范

## 4.1 页面视觉结构

根据当前 UI 设计稿，整体界面采用“单屏手机卡片 + 剧情节点连线”的方式组织。每个剧情状态以一个竖向手机画框呈现，节点之间通过连线表达跳转关系。

| 模块 | 设计表现 | 开发说明 |
|---|---|---|
| 页面容器 | 竖向手机屏幕卡片，黑色描边，圆角较大 | 小程序单页可使用固定安全区容器；建议模拟手机卡片视觉 |
| 角色主体 | 黑白小毛咪插画，占据页面上半区或中部 | 每个状态可绑定不同猫咪姿态图，如站立、坐着、趴着、尾巴动、伸手等 |
| 文案按钮 | 页面底部纵向排列胶囊按钮 | 对应状态机中的 `OPTION_VALID` / `OPTION_INVALID` 节点 |
| 节点关系 | 设计稿以横向/纵向连线表现剧情流转 | 代码中不必渲染全部连线，实际小程序只展示当前状态即可 |
| 结果弹窗 | 独立卡片弹窗，包含图像、结果文案、按钮 | 对应 `RESULT_MODAL` 节点 |
| 失败弹窗 | 文案强调“可恶！”、“哎呀！被亲亲了！”、“重选一次吧” | 对应无效点击反馈弹窗 |
| 成功分享弹窗 | 显示 `耶斯！小毛咪不生气啦！` 与 `截图分享给姐姐` | 对应 `END_SUCCESS_SHARE` |

## 4.9 UI 到状态机的开发映射原则

1. 首页 `SCENE_001` 只渲染一个手机画框：主标题为 `这是谁家的小毛咪在路边？`，底部为 3 个主入口按钮。
2. 首页 3 个主入口按钮分别进入：摸摸互动组、投喂互动组、生气安抚组。
3. 设计稿中的每一个手机画框，对应一个 `SCENE` 或 `PROMPT` 状态。
4. 设计稿中的每一个底部胶囊按钮，对应一个 `OPTION_VALID` 或 `OPTION_INVALID` 节点。
5. 设计稿中的结果卡片，对应一个 `RESULT_MODAL` 节点。
6. 设计稿中的“截图分享给姐姐”按钮，对应 `createSharePoster` 行为。
7. 设计稿中的“重选一次吧”按钮，对应 `returnToParentPrompt` 行为。
8. 设计稿中的“是/否”大按钮，应作为独立确认页组件处理，而不是普通列表按钮。
9. `哼！` 与 `哼。` 不作为首页入口，它们属于 `咦 这个小毛咪看起来在生气` 之后的二级分支。

## 10. 状态流转核心逻辑建议

```ts
function handleNodeTap(node) {
  if (!node.isClickable && node.nodeType === 'OPTION_INVALID') {
    showModal(node.triggerFeedback);
    return;
  }

  if (node.triggerFeedback?.type === 'audio') {
    playAudio(node.triggerFeedback.audioId);
  }

  if (node.triggerFeedback?.type === 'modal') {
    showModal(node.triggerFeedback);
  }

  if (node.nextId && node.nextId !== 'END' && node.nextId !== 'END_SUCCESS_SHARE') {
    setCurrentNode(node.nextId);
  }
}
```

# 11. 当前版本待确认项

| 待确认项 | 原因 | 建议 |
|---|---|---|
| 主入口是否一次性展示 5 个章节 | 原框架是剧情树文本，未说明首页导航方式 | 首页可展示 5 个入口：摸摸、投喂、生气、哼哼、麻袋 |
| “不可点击”是否需要视觉置灰 | 原文同时写了不可点击和点击弹窗 | 建议置灰但保留点击反馈 |
| 火锅“吆喝”是否需要音频 | 原文未明确是否播放声音 | 作为可选音效素材处理 |
| “比划”“指指”是否需要动效 | 原文偏动作描述 | 可做轻量动效，也可仅作为文案展示 |
| 普通终点是否返回首页 | 原文未说明 | 默认弹窗关闭后停留当前页，另设返回首页按钮可选 |
| 是否按设计稿渲染全量剧情树连线 | UI 设计稿展示了完整流程关系，但小程序实际不一定需要全图导航 | 建议开发版只渲染当前手机状态；完整流程图仅用于设计/开发说明 |
| 成功弹窗里的感叹号是否统一 | UI 稿为“耶斯！小毛咪不生气啦！”；原剧情中部分无感叹号 | 建议统一使用 UI 稿版本 |
| 失败弹窗是否统一加入“可恶！” | UI 稿中失败弹窗出现“可恶！” | 建议作为失败弹窗的强化标题 |
